Learning Stories (Protocol & Concurrency Focus)

Goal: Provide learning-focused user stories that fit directly into the development plan so you can pick concrete, low-level tasks that both ship value and cement protocol/concurrency skills.

How to use: - Pick a story from your current Phase/Epic that also aligns with the Protocol Curriculum sprint. - Open an issue using the Protocol Lab template and label it with the protocol (e.g., protocol:messaging) and sprint (e.g., sprint:2). - Implement the story end-to-end and add a short note under docs/stories/ linking the PR and what you learned.

Sprint 1 — REST + Foundations

Story L1.1 — Product Create/Read API (REST) - Fit: Phase 4 (Product Service), EPIC-003 - Services: product-service - Protocol focus: REST (OpenAPI-driven) - Concurrency: Reactive (WebFlux) or MVC (Virtual Threads later) - Acceptance: - [ ] OpenAPI with POST /products, GET /products/{id} - [ ] Validation & consistent error responses (bv-common-exceptions) - [ ] Unit + integration tests (containerized Postgres) - [ ] Ops: traces/metrics/logs present - Docs: Update API examples in 03-DEVELOPMENT/microservices-patterns.md

Story L1.2 — Auth Login Flow (REST) - Fit: Phase 3 (Auth), EPIC-002 - Services: bv-auth-service - Protocol focus: REST - Acceptance: - [ ] OpenAPI covers /auth/login, /auth/register - [ ] BCrypt, basic RBAC, rate limiting - [ ] Tests for success/failure paths - [ ] Observability wired

Sprint 2 — Messaging (Kafka) + Contracts

Story L2.1 — OrderCreated → Inventory Adjustment (Messaging) - Fit: Phase 6, EPIC-005 (Multi-Domain foundation) - Services: order-service (producer), inventory-service (consumer) - Protocol focus: Messaging, Event Contracts - Acceptance: - [ ] Envelope fields set; events versioned - [ ] Idempotent consumer with retries/backoff per ADR-006 - [ ] Integration tests with Kafka - [ ] Sequence diagram added in API Styles Track (Messaging)

Story L2.2 — Event Contract Governance - Fit: Phase 6, aligns with ADR-003 - Services: event contracts folder only - Protocol focus: Contract-first, versioning - Acceptance: - [ ] CHANGELOG updated; breaking changes flagged - [ ] Lint/check script (optional) for schema shape

Sprint 3 — Concurrency Scenarios (Reactive vs Virtual Threads)

Story L3.1 — Reactive Checkout (Scenario A) - Fit: Phase 7, EPIC-003/Payments integration - Services: order-service, payment-adapter-service - Protocol focus: REST + non-blocking I/O; Concurrency: Reactive (WebFlux) - Acceptance: - [ ] Reactive composition (WebClient + R2DBC) - [ ] Backpressure test and resilience (timeouts/retries/circuit breaker) - [ ] Trace propagation across calls

Story L3.2 — Virtual Threads Webhook Dispatcher (Scenario C) - Fit: Phase 7, EPIC-005 - Services: partner-webhook-dispatcher - Protocol focus: Webhooks; Concurrency: Virtual Threads (Java 21) - Acceptance: - [ ] VT executor with bounded concurrency - [ ] HMAC verification; retries; error handling - [ ] Tracing for each webhook attempt

Sprint 4 — GraphQL + BFF Reads

Story L4.1 — Read Aggregation via GraphQL - Fit: (Future) BFF layer, EPIC TBD - Services: BFF (new), reading from product/order - Protocol focus: GraphQL - Acceptance: - [ ] Query orderById, ordersByStatus with batching to avoid N+1 - [ ] Observability: log/trace resolvers and downstream calls

Sprint 5 — Real-time Updates (SSE/WebSockets)

Story L5.1 — Inventory SSE - Fit: Phase 7 (Streaming Inventory) - Services: inventory-service - Protocol focus: SSE - Acceptance: - [ ] Flux endpoint; reconnect/heartbeats - [ ] Load test basic scenario

Story L5.2 — Notifications via WebSockets - Fit: Phase 7 - Services: notification-service - Protocol focus: WebSockets - Acceptance: - [ ] Authenticated sessions; broadcast and targeted messages - [ ] Basic soak test

Sprint 6 — Batch & Replay

Story L6.1 — Replay CSV → Kafka - Fit: Phase 7, EPIC-005 - Services: replay-service - Protocol focus: Batch/File + Messaging - Concurrency: Virtual Threads recommended - Acceptance: - [ ] Batch replay with checkpoints and backoff - [ ] Deduplication/idempotency
