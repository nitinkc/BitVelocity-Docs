Actionable Build Plan — Start to Finish

Purpose: Provide a concrete, low-level, start-to-finish build sequence you can pick up and execute. Each task has a clear outcome, acceptance checks, and the exact repo areas to touch. Use this as the backbone for Sprint Planning and daily/hourly breakdowns.

Assumptions - Monorepo with Java services (Maven), Spring Boot/WebFlux, and supporting infra (Postgres, Redis, Kafka) - Local dev on Docker Desktop; docs site in this repo - Protocol-first: every change starts with a contract and ends with documentation and traceability

How to use - Work top-to-bottom in order; stop at each gate and validate acceptance - Each task can be split into sub-tasks day-by-day; keep PRs small and focused - Label issues/PRs using the provided labels to maintain traceability

Phase 0 — Repo & Docs Groundwork (0.5–1 day)

1) Create “bootstrap” docs entrypoint - Files: docs/stories/QUICK-START.md (if missing), update docs/index.md - Acceptance: - [ ] Quick Start page exists and links to this plan, Developer Plan, Protocol Curriculum - [ ] mkdocs nav includes Quick Start - Notes: Helps new contributors orient fast

2) Verify docs navigation & site deploy - Files: mkdocs.yml, docs/index.md - Acceptance: - [ ] mkdocs serve runs locally without errors - [ ] mkdocs gh-deploy completes (CI or local)

Phase 1 — Build System & Shared Foundations (2–3 days)

1) Parent POM and BOM - Modules: bv-core-parent, bv-core-platform-bom - Acceptance: - [ ] Parent POM defines plugin mgmt and shared config - [ ] BOM pins dependency versions - [ ] At least one service builds off parent with BOM imported

2) Shared libraries skeleton - Modules: bv-core-common (+ submodules listed in Projects & Modules) - Acceptance: - [ ] bv-common-entities, bv-common-events, bv-common-exceptions, bv-common-logging, bv-common-auth, bv-common-security compile - [ ] Basic unit test per module (one assertion) - [ ] README in each module documents purpose and usage

3) CI sanity pipeline - Files: .github/workflows/build.yml - Acceptance: - [ ] Build on PR for changed modules - [ ] Fails on test or style errors

Phase 2 — Local Infrastructure (1–2 days)

1) Docker Compose stack - Files: docker-compose.yml - Services: Postgres, Redis, Kafka+ZK, Schema Registry (optional) - Acceptance: - [ ] docker compose up -d brings stack up - [ ] Health checks for each service pass

2) Kind (optional initial) or K8s manifests scaffold - Path: k8s/ - Acceptance: - [ ] Base namespace and shared secrets examples - [ ] Service template manifests checked in (deployment, svc, configmap)

Phase 3 — Auth Service (2–4 days)

1) Contract-first API for auth - Files: auth-service/openapi/auth.yaml - Acceptance: - [ ] OpenAPI describes register/login/refresh/me endpoints - [ ] Error model aligns with bv-common-exceptions

2) Implement bv-auth-service - Module: bv-auth-service - Acceptance: - [ ] Register, login, JWT issuance/validation - [ ] BCrypt hashing, basic RBAC, rate limit on login - [ ] Unit + integration tests (containerized Postgres) - [ ] Observability basics: traces, metrics, structured logs

3) Docs & examples - Files: docs/03-DEVELOPMENT/microservices-patterns.md (Auth section), docs/stories/ - Acceptance: - [ ] Endpoint examples and curl snippets documented - [ ] Link from Protocol Curriculum where relevant

Phase 4 — Product Service (2–4 days)

1) Contract-first API for products - Files: product-service/openapi/product.yaml - Acceptance: - [ ] CRUD operations, validation, error model

2) Implement product-service - Module: bv-eCommerce-core/product-service - Acceptance: - [ ] Entities, migrations, repository, service, controller - [ ] Request validation; consistent error responses - [ ] Unit + integration tests (containerized Postgres)

3) Docs & Postman/Insomnia collection - Files: docs/03-DEVELOPMENT/microservices-patterns.md (REST), docs/stories/ - Acceptance: - [ ] REST usage examples; collection committed

Phase 5 — Observability Baseline (1–2 days)

1) Wire OpenTelemetry and Micrometer - Modules: auth, product (as exemplars) - Acceptance: - [ ] Trace IDs flow through incoming/outgoing calls - [ ] Metrics endpoint exposes default + custom counters

2) Grafana dashboards and alerting (basic) - Files: grafana/ dashboards JSON; docs/04-OPERATIONS/observability.md - Acceptance: - [ ] Dashboards for latency, error rate, throughput - [ ] Basic alerts on error spikes

Phase 6 — Messaging & Event Contracts (2–4 days)

1) Define event envelope & contracts - Files: docs/event-contracts/schema/, docs/event-contracts/ecommerce/ - Acceptance: - [ ] Envelope fields set (id, type, version, ts, traceId) - [ ] Key ecommerce events (OrderCreated, InventoryAdjusted) defined - [ ] CHANGELOG updated

2) Publish/consume in services - Modules: order-service, inventory-service - Acceptance: - [ ] OrderCreated published; Inventory service consumes and updates stock - [ ] Idempotency and retries per ADR-006 - [ ] Integration tests with Kafka

3) Docs - Files: docs/stories/API-STYLES-TRACK.md (Messaging section updates) - Acceptance: - [ ] Sequence diagrams and usage documented

Phase 7 — Reactive vs Virtual Threads Scenarios (3–5 days)

1) Reactive Checkout (A) - Modules: order-service, payment-adapter-service - Acceptance: - [ ] Non-blocking REST calls, R2DBC, backpressure - [ ] Resilience config (timeouts, retries, circuit breakers) - [ ] Reactor test for backpressure

2) Streaming Inventory (B) - Modules: inventory-service, notification-service - Acceptance: - [ ] SSE from inventory; WebSocket notifications - [ ] Load test basic scenario

3) Virtual Threads Webhooks (C) - Module: partner-webhook-dispatcher - Acceptance: - [ ] VT executor; bounded concurrency; tracing

4) Virtual Threads Replay (D) - Module: replay-service - Acceptance: - [ ] Batch replay with VT; backoff and checkpoints

5) Docs - Files: docs/stories/JAVA-CONCURRENCY-TRACK.md - Acceptance: - [ ] A/B comparison and lessons learned captured

Phase 8 — CI/CD and K8s (2–4 days)

1) Build and test pipelines - Files: .github/workflows/build.yml, .github/workflows/test.yml - Acceptance: - [ ] On PR, build touched modules, run tests, generate site

2) Container images & SBOM - Files: Dockerfile per service, SBOM generation step - Acceptance: - [ ] Images build; SBOM attached/artifacts uploaded

3) K8s deploy - Path: k8s/ - Acceptance: - [ ] Dev namespace deploy job; smoke tests

Phase 9 — Hardening & Polish (2–3 days)

1) Security - Acceptance: - [ ] Secrets management, TLS (local/dev via mkcert) - [ ] Basic authZ policies enforced

2) Performance & cost - Acceptance: - [ ] Baseline load tests; simple cost estimates

3) Docs & ADRs - Acceptance: - [ ] ADRs updated; docs reflect final state

Traceability & Labels

  • Open an issue per task using “Protocol Lab” or a new “Build Task” template
  • Apply labels:
  • Area (area:docs, area:site, area:event-contracts, service area when added)
  • Protocol (if relevant)
  • Sprint (sprint:1..sprint:12)
  • Track (track:curriculum, etc.)
  • Type (type:lab, type:enhancement)

Daily Execution Pattern

  • Morning: pick the next task, confirm acceptance, define a tiny PR scope
  • Midday: implement + tests; draft docs updates alongside
  • Afternoon: run tests and local validations; open PR with checklists
  • End of day: update issue with notes, blockers, next steps
